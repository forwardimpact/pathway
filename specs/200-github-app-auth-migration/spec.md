# Spec 200: GitHub App Authentication Migration

## Status

Draft

## Problem

The continuous improvement system (six scheduled workflows, four agent personas)
authenticates to GitHub using a fine-grained Personal Access Token (PAT) stored
as `CLAUDE_GH_TOKEN`. This creates three problems:

1. **Token expiry.** Fine-grained PATs expire after at most one year. When the
   token expires, all six agent workflows silently fail until a maintainer
   notices, rotates the token, and updates the repository secret. There is no
   automated warning before expiry.

2. **Individual identity.** The PAT is tied to a specific GitHub user account.
   Commits and API calls made by the agents appear as that individual engineer,
   not as the automation system. This obscures the audit trail — it is unclear
   whether a push, merge, or comment came from a human or an agent. It also
   means the individual's account must remain active and retain the necessary
   permissions.

3. **Adoption friction for downstream installations.** The Forward Impact
   products are designed for external consumption (see CLAUDE.md § Distribution
   Model). Organizations that fork or install the monorepo must create their own
   PAT, understand the required permission scopes, and manage rotation
   themselves. A pre-configured GitHub App that anyone can install with one
   click removes this friction and provides a standardized identity for CI
   agents across all installations.

## Scope

### In scope

- Create a public GitHub App owned by the Forward Impact organization that
  provides the permissions the CI agents need.
- Migrate all six agent workflows and the shared composite action from
  `CLAUDE_GH_TOKEN` (PAT) to installation tokens generated by the GitHub App.
- Update the composite action's git identity to use the App's bot account
  instead of `github-actions[bot]`.
- Document the App setup for downstream installations (install the pre-built
  App, or create your own).
- Update `CONTINUOUS_IMPROVEMENT.md` and `CONTRIBUTING.md` to reflect the new
  authentication model.

### Out of scope

- Changing what the agents do (prompts, skills, agent profiles).
- Changing workflow schedules, concurrency groups, or timeout values.
- Migrating non-agent workflows (check-quality, check-test, publish-\*,
  website).
- Revoking the existing PAT (the repository owner does that manually after
  migration is verified).

## Success Criteria

1. All six agent workflows authenticate using a GitHub App installation token
   instead of a PAT.
2. Commits made by agents show the GitHub App's bot identity, not a personal
   account.
3. Wiki clone/push in the composite action works with the App token.
4. No workflow uses the `CLAUDE_GH_TOKEN` secret.
5. A public GitHub App exists that downstream installations can install with one
   click to get the same CI agent pattern working.
6. Documentation describes both the pre-built App (one-click install) and the
   self-hosted App option for organizations that prefer their own.
7. The `security-audit` workflow continues to operate with `contents: read` only
   (least privilege preserved).
