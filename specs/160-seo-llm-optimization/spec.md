# 160 — SEO and LLM Agent Optimization

## Problem

The Forward Impact website (www.forwardimpact.team) has no `sitemap.xml`, no
`robots.txt`, no per-page markdown representations, and no `llms.txt` file. This
means:

1. **Search engines** cannot efficiently discover or prioritize pages. Crawlers
   rely on following links from the home page, which is slow and incomplete for
   deeper documentation pages. Without a sitemap, new or updated pages may take
   weeks to appear in search results.

2. **LLM agents** that visit the site to answer user questions must parse complex
   HTML with navigation, styling, and scripts to extract content. This is
   imprecise and wastes context window tokens on boilerplate. The llms.txt
   standard (https://llmstxt.org) exists specifically to solve this — providing a
   curated, markdown-formatted entry point that LLMs can consume directly.

3. **Per-page markdown** does not exist in the built output. When an LLM agent
   or downstream tool needs the content of a single page in clean markdown, the
   only option is to scrape and reverse-engineer the HTML. Since libdoc already
   has the markdown source, producing a `.md` companion for each HTML page is
   nearly free.

libdoc currently builds markdown to HTML and copies static assets. It has no
awareness of the full page inventory and produces no machine-readable discovery
files. The website has no manually curated LLM entry point either.

## Scope

### In scope

- **libdoc: sitemap.xml generation** — Automatically produce a `sitemap.xml` at
  the site root during `build()`, listing every HTML page with its URL.

- **libdoc: per-page markdown output** — For each HTML page produced, also write
  a clean `.md` file at a parallel path (e.g., `/docs/map/index.md` alongside
  `/docs/map/index.html`). This is the post-front-matter, pre-template markdown
  content — the page body without YAML metadata or HTML wrapper.

- **website: llms.txt** — A manually curated file in `website/public/llms.txt`
  following the llms.txt specification (https://llmstxt.org). This file is
  authored and maintained by hand, not generated by libdoc.

- **website: robots.txt** — A `website/public/robots.txt` that references the
  sitemap location.

- **Skill updates** — Update the `libs-web-presentation` and `website` Claude
  skills to document the new outputs and conventions.

### Out of scope

- Open Graph / social meta tags
- JSON-LD structured data / schema.org markup
- RSS or Atom feeds
- Search indexing or full-text search
- Automated llms.txt generation from libdoc
- Changes to the HTML template or visual design

## Changes

### 1. libdoc: sitemap.xml generation

During `build()`, after all pages are processed, DocsBuilder produces a
`sitemap.xml` in the output root. The sitemap follows the Sitemaps protocol
(sitemaps.org):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://www.forwardimpact.team/</loc>
  </url>
  <url>
    <loc>https://www.forwardimpact.team/about/</loc>
  </url>
  <!-- one <url> entry per generated HTML page -->
</urlset>
```

The base URL is provided as a parameter to `build()` (or via a new front matter
/ config mechanism). Each generated HTML page gets a `<loc>` entry using its
clean URL path (directory-style, trailing slash).

No `<lastmod>`, `<changefreq>`, or `<priority>` elements — keep it minimal.
These are optional in the protocol and add maintenance burden without meaningful
benefit for a site of this size.

### 2. libdoc: per-page markdown output

For each page built from a `.md` source, DocsBuilder writes a companion markdown
file alongside the HTML output. The markdown file contains the page body only —
no YAML front matter, no HTML template wrapper.

Output path mapping:

| HTML output              | Markdown output          |
| ------------------------ | ------------------------ |
| `index.html`             | `index.md`               |
| `about/index.html`       | `about/index.md`         |
| `docs/map/index.html`    | `docs/map/index.md`      |
| `docs/model/core/index.html` | `docs/model/core/index.md` |

The markdown content is the source markdown after front matter extraction — the
same content that gets passed to the marked parser. No transformation or
re-serialization from HTML. This keeps the output faithful to the source and
avoids round-trip quality loss.

Internal links in the markdown output should use the same transformed paths as
the HTML (directory-style URLs, not `.md` references), so the markdown files
work as standalone documents that link to the live site.

### 3. website: llms.txt

A new file at `website/public/llms.txt` provides an LLM-optimized entry point
for the site. This file follows the llms.txt specification
(https://llmstxt.org) and is **manually curated** — not generated by libdoc.

The file is placed in `website/public/` so that libdoc copies it to the output
root (the existing `public/` → root copy behavior).

#### Structure

```markdown
# Forward Impact

> Engineering framework platform. Define what good engineering looks like,
> generate career paths, AI agent profiles, and team capability analysis.
> Six products: Map, Pathway, Basecamp, Guide, Landmark, Summit.

Forward Impact helps engineering organizations define skills, levels, and
behaviours in a shared framework that humans and AI agents both use. The
data model is YAML-based and drives all products.

## Products

- [Map](/map/index.md): Data product — define and validate engineering frameworks
- [Pathway](/pathway/index.md): Career paths, job definitions, and agent profiles
- [Guide](/guide/index.md): AI agent for framework-aware developer guidance
- [Basecamp](/basecamp/index.md): Personal ops center with knowledge graph and scheduling
- [Landmark](/landmark/index.md): Engineering signal analysis and evidence views
- [Summit](/summit/index.md): Team capability coverage and staffing scenarios

## Documentation

- [Data Model](/docs/model/index.md): Core formula — Discipline x Track x Level
- [Core Entities](/docs/model/core.md): Skills, capabilities, behaviours, stages
- [Engineering Lifecycle](/docs/model/lifecycle.md): Six-stage delivery lifecycle
- [Map Data Product](/docs/map/index.md): Schema, validation, YAML structure
- [Pathway CLI & Web](/docs/pathway/index.md): Formatters, templates, static site
- [Agent Generation](/docs/pathway/agents.md): AI agent profile derivation
- [CLI Reference](/docs/pathway/reference.md): File organization and commands
- [Synthetic Data](/docs/universe/index.md): Universe DSL for test data generation
- [Landmark Analysis](/docs/landmark/index.md): Evidence views and practice patterns
- [Basecamp Operations](/docs/basecamp/index.md): Scheduler, knowledge base, CLI

## Schema

- [JSON Schema](/schema/json/): Machine-readable framework validation schemas
- [RDF/SHACL](/schema/rdf/): Semantic web representations

## Optional

- [About](/about/index.md): Philosophy and organizational context
```

#### Content guidelines

The llms.txt file should cover:

- **Project identity** — What Forward Impact is, in one paragraph. Mention the
  six products, the YAML-based data model, and the dual human/agent audience.
- **Product pages** — One link per product with a concise description of what it
  does and who it serves. Links point to the per-page markdown files (not HTML).
- **Documentation** — Links to each documentation page's markdown file. Brief
  descriptions focused on what an LLM would need to know to answer questions
  about the framework.
- **Schema** — Links to the JSON Schema and RDF/SHACL directories so LLMs can
  discover machine-readable definitions.
- **Optional section** — Secondary pages (like About) that provide context but
  are not essential for understanding the product.

All links in llms.txt point to the `.md` companion files (not `.html`), so that
LLMs retrieving linked content get clean markdown rather than HTML.

This file must be reviewed and updated whenever pages are added or removed from
the website. It is a curated summary, not an exhaustive index.

### 4. website: robots.txt

A new file at `website/public/robots.txt`:

```
User-agent: *
Allow: /

Sitemap: https://www.forwardimpact.team/sitemap.xml
```

This tells crawlers where to find the sitemap. The `Allow: /` is explicit but
redundant (default behavior) — included for clarity.

## Success Criteria

1. `npx fit-doc build --src=website --out=dist` produces:
   - `dist/sitemap.xml` with a `<url>` entry for every HTML page
   - A `.md` file alongside every `.html` file in the output
   - `dist/llms.txt` copied from `website/public/llms.txt`
   - `dist/robots.txt` copied from `website/public/robots.txt`

2. `dist/sitemap.xml` is valid XML conforming to the Sitemaps protocol

3. Each `.md` output file contains the page body in clean markdown with
   directory-style internal links

4. `dist/llms.txt` follows the llms.txt specification with an H1, blockquote,
   H2-delimited link sections, and an Optional section

5. All links in `llms.txt` resolve to existing `.md` files in the built output

6. The `libs-web-presentation` and `website` skills accurately describe the new
   outputs and conventions

7. Existing tests pass; new tests cover sitemap generation and markdown output
