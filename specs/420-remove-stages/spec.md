# 420 — Remove stages from the engineering pathway

**Status:** review **Author:** D. Olsson **Created:** 2026-04-13

## Problem

Stages model the software development lifecycle as a fixed sequence of phases
(specify, plan, scaffold, code, review, deploy). Every agent is generated as a
discipline x track x stage combination, producing names like `se-platform-code`
and `se-platform-review`. This creates three problems:

1. **Narrow modeling.** Real engineering work does not follow a strict linear
   pipeline. An engineer reviews code while planning the next iteration, deploys
   while specifying a hotfix, scaffolds while coding. Forcing stage boundaries
   onto agents constrains them to an artificial slice of engineering activity.

2. **Combinatorial explosion.** The Cartesian product of disciplines x tracks x
   stages generates many agents that differ only in which stage constraints and
   checklists they carry. With 5 disciplines, ~3 tracks each, and 6 stages the
   matrix produces ~90 agent profiles when 15 would suffice.

3. **Misplaced checklists.** Stage-level checklists and handoffs live in
   `stages.yaml` and in each skill's `agent.stages.{id}` block. These duplicate
   or conflict with skill-level guidance. Checklists are properties of skills,
   not of lifecycle phases.

## Proposed solution

Remove stage as a first-class concept from the pathway product, libskill,
framework data schema, and synthetic data pipeline. Agents become one per
discipline (with optional track modifier), carrying the full skill matrix for
that discipline without stage scoping.

This is a clean break. No backward compatibility is provided — no deprecation
period, no migration command, no shims. External users must update their
framework data to the new format before upgrading. `fit-map validate` will
reject `stages.yaml` and any `agent.stages` nesting in capability files.

### Agent identity after removal

| Before (stage-scoped) | After (stage-free)            |
| --------------------- | ----------------------------- |
| `se-platform-code`    | `software-engineer--platform` |
| `se-platform-review`  | _(same agent)_                |
| `se-sre-plan`         | `software-engineer--sre`      |
| `de-null-scaffold`    | `data-engineer`               |
| `em-null-specify`     | `engineering-manager`         |

Pattern: `{kebab(roleTitle)}` or `{kebab(roleTitle)}--{track}` when a track is
set. The discipline's `roleTitle` field already provides the human-readable
name.

### Agent profile content

Each agent profile includes:

- **Identity** from `discipline.agent.identity` + `track.agent.identity`
- **Priority** from discipline + track
- **Constraints** from discipline + track
- **Full skill matrix** — all skills for that discipline at their proficiency
  levels, not filtered by stage
- **Skill checklists** — each skill's own `readChecklist` and `confirmChecklist`
  (flat per skill, not nested under stages)
- **Behaviours** with modifiers from discipline + track

No stage constraints, stage handoffs, stage return formats, or stage
transitions.

### Skill data after removal

Each skill currently nests checklists under `agent.stages.{stageId}`. After
removal, each skill carries a single flat `agent.focus`, `agent.readChecklist`,
and `agent.confirmChecklist`. Consolidated checklists must follow the guidelines
in [CHECKLISTS.md](../../CHECKLISTS.md) (READ-DO for `readChecklist`, DO-CONFIRM
for `confirmChecklist`, 5–9 killer items, precise actionable phrasing).

### Synthetic data pipeline

All pathway data under `data/pathway/` is generated by the synthetic data DSL
and libraries. The pipeline must be updated so that regenerating data produces
the new stage-free format: no `stages.yaml` output, no stage parsing in the DSL,
and capability prompts producing flat
`agent.{focus, readChecklist, confirmChecklist}` per skill instead of
`agent.stages.{stageId}.*`.

## Scope

### In scope

- **Framework data** — Delete stage definitions (`stages.yaml`), stage schemas
  (JSON Schema, RDF), and flatten `agent.stages` nesting in capability YAML to
  single `agent.focus` / `agent.readChecklist` / `agent.confirmChecklist` per
  skill.
- **Map product** — Remove stage schemas, stage validation rules, stage loading,
  and stage ordering utilities.
- **Pathway product** — Remove the `stage` CLI subcommand, `--stage` /
  `--all-stages` flags from the `agent` command, stage page and formatters,
  stage selector in the web UI agent builder, and stage references on the
  landing page. Update `build-packs` to generate one agent per discipline/track.
- **libskill** — Remove stage-based agent generation, rewrite agent derivation
  to produce one agent per discipline (x track), and derive checklists from flat
  skill data instead of stage-keyed blocks.
- **Synthetic data pipeline** — Remove stage parsing from the DSL, stage
  vocabulary, stage prompt templates, and stage rendering. Regenerated data must
  produce the new stage-free format.
- **Documentation** — Update CLAUDE.md domain concepts to remove stages.

### Out of scope

- **Summit and Landmark** — Neither product references stages, stage IDs, or
  stage ordering in their source or data (Summit models team capability via
  skill matrices, not lifecycle phases).
- **Guide** — Guide consumes agent profiles but is stage-agnostic; it will
  receive the new simpler profiles without code changes.
- **Basecamp** — No stage dependency.
- **Adding new agent capabilities** — This spec only removes stages; enriching
  agent profiles is future work.
- **Renaming disciplines or tracks** — Orthogonal concern.

## Migration

Internal framework data is migrated as part of the implementation: each skill's
`agent.stages.*` blocks are flattened into a single `agent.focus`,
`agent.readChecklist`, and `agent.confirmChecklist`. The consolidated checklists
must comply with [CHECKLISTS.md](../../CHECKLISTS.md).

### External user impact

This is a breaking change with no backward compatibility. External users who
have defined `stages.yaml` or use `agent.stages` nesting in capability files
must manually update their framework data before upgrading. The next major
version of `fit-map` and `fit-pathway` must:

- Remove stage schema from published schemas.
- Document the required manual migration steps in release notes (delete
  `stages.yaml`, flatten `agent.stages.*` to
  `agent.{focus, readChecklist, confirmChecklist}` per skill).

## Success criteria

1. `bunx fit-map validate` passes with no stage-related schemas or data.
2. `bunx fit-pathway agent software_engineering --track=platform` produces a
   single agent profile named `software-engineer--platform` with the full skill
   matrix.
3. `bunx fit-pathway agent software_engineering` (no track) produces
   `software-engineer`.
4. `bunx fit-pathway stage` exits with an unknown-command error.
5. The pathway web UI agent builder has no stage selector and shows one agent
   per discipline/track combination.
6. `bunx fit-pathway build-packs` generates packs with one agent per
   discipline/track.
7. All existing tests pass or are updated to reflect the new structure.
8. `grep -r 'stageId\|stageOrder\|agent-stage\|getStageOrder\|compareByStageOrder' libraries/ products/`
   returns no matches.
9. Regenerating synthetic data (`just synth` or equivalent) produces
   `data/pathway/` without `stages.yaml` and with flat skill checklists.
10. `STAGE_NAMES` is absent from libsyntheticgen exports.
11. Consolidated skill checklists comply with CHECKLISTS.md: READ-DO semantics
    for `readChecklist`, DO-CONFIRM for `confirmChecklist`, 5-9 items each,
    precise actionable phrasing.
