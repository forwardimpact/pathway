# Spec 820 — Synthetic pipeline uniform activity interface

## Problem

The synthetic data pipeline that backs `fit-terrain` (the tool Platform
Builders hire to generate synthetic data for development, testing, and demos —
[JTBD § Evaluate and Improve Agents](../../JTBD.md)) is wired
type-by-type with no shared contract. Each "activity type" (GitHub commit,
GitHub PR, GitHub review, GetDX snapshot comment, GetDX webhook) is plumbed
through four libraries by ad-hoc functions whose signatures, context shapes,
and registration points all differ. The gap is observable: the same DSL fact
that the LLM is told for one activity type is dropped on the floor for
another.

**Concrete evidence — comment prose drops compound driver impact.**
The DSL models `dx_drivers` per team per scenario with `trajectory` and
`magnitude`. Webhook PR/review prose generation passes the full driver array
into the LLM context; snapshot-comment prose generation passes only a single
driver scalar.

| Activity type           | File                                                                  | Driver context shape passed to LLM                                                |
| ----------------------- | --------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| PR body (webhook)       | `libraries/libsyntheticgen/src/engine/prose-keys.js:210-222`          | `drivers: wk.drivers` (full array of `{driver_id, trajectory, magnitude}`)        |
| Review body (webhook)   | `libraries/libsyntheticgen/src/engine/prose-keys.js:225-237`          | `drivers: wk.drivers` (full array)                                                |
| Snapshot comment        | `libraries/libsyntheticgen/src/engine/prose-keys.js:120-140`          | `driver`, `direction`, `magnitude` (one driver only — no array, no other drivers) |

Result: when a comment is generated by a person on a team that the active
scenario marks as declining on `clear_direction`, `deep_work`, **and**
`code_review`, the LLM is told only about the single primary driver. Other
co-declining drivers — and the magnitudes that rank them — never reach the
prompt. The prose template at
`libraries/libsyntheticprose/src/prompts/prose-user.prompt.md` has a
conditional `driverContext` block that webhook keys populate and comment keys
leave empty.

**Concrete evidence — no shared interface across the five activity types.**
`activity.js` wires sub-generators by hand
(`libraries/libsyntheticgen/src/engine/activity.js:73,82,98,102`); prose-key
collectors are each their own function called from a hand-written if-chain
(`libraries/libsyntheticgen/src/engine/prose-keys.js:147-198`); the renderer
is a list of per-type functions invoked from
`renderRawDocuments` (`libraries/libsyntheticrender/src/render/raw.js:17-27`).
None of the three lists agrees on a name, a signature, or an order. Adding
the sixth activity type requires editing all three lists plus the prompt
surface and a validator, with no contract telling the contributor what shape
each expects.

The DSL is rich (drivers with trajectory and magnitude, scenarios, narratives,
phases, project topics, team membership, person level and discipline) but
each activity type chooses its own subset of that information arbitrarily.
The choice is a per-function judgement call, not a documented decision, and
the comment-vs-webhook asymmetry above is one instance of it.

## Goal

Replace the ad-hoc, type-by-type plumbing of the synthetic activity pipeline
with a single uniform contract that every activity type implements. The
contract names what each activity type contributes at each stage of the
pipeline (deterministic generation, prose-context construction, rendering,
validation) and what shape the DSL-derived domain context (drivers, scenario,
team, role, project) takes when it crosses into the LLM. As a consequence of
the contract being applied uniformly, snapshot-comment prose receives the
full driver context that webhook prose already receives, and the asymmetry
above disappears. Adding the next activity type becomes a single registration
of one implementation, not a coordinated edit across four libraries.

## Scope (in)

- **The five existing activity types** that today live as ad-hoc per-type
  functions in `libsyntheticgen` and `libsyntheticrender`: GitHub commits,
  GitHub PRs, GitHub PR reviews, GetDX snapshot comments, GetDX webhooks. Each
  must be expressible through one shared activity-type contract.
- **The four pipeline touchpoints** every activity type has today, which the
  contract must name: deterministic tier-0 generation
  (`libsyntheticgen/src/engine/activity*.js`), prose-context construction
  (`libsyntheticgen/src/engine/prose-keys.js`), output rendering
  (`libsyntheticrender/src/render/raw.js`), and per-type validation
  (`libsyntheticrender/src/validate-activity.js`).
- **The prose-context schema for activity prose.** A single named shape for
  the domain context that crosses into the LLM, covering at minimum the DSL
  facts the activity author can already name (scenario, team, role, full
  driver array with trajectory and magnitude, project topic and tone where
  relevant). Snapshot comments must populate the full driver array, not a
  single-driver scalar.
- **The prompt-side consumer of that schema.** The user prompt at
  `libraries/libsyntheticprose/src/prompts/prose-user.prompt.md` (and any
  templates that share its surface) must render the schema's driver context
  for every activity type that supplies it, including snapshot comments.
- **Unit tests for the logic.** Per-activity-type unit tests covering the
  contract-required outputs (generated entities, prose-context entries,
  rendered output keys, validation outcomes) for at least one
  representative case per activity type, including a snapshot comment whose
  author is on a team with multiple co-declining drivers and whose
  prose-context entry contains the full driver array.

## Scope (out)

- **The four-library boundary** (`libterrain`, `libsyntheticgen`,
  `libsyntheticprose`, `libsyntheticrender`). Whether the contract preserves,
  redraws, or eliminates the boundary is a design question, not a spec
  question. The spec requires only that the contract exist and be applied
  uniformly.
- **The DSL grammar and parser.** The terrain DSL today already names every
  domain fact this spec needs — drivers, trajectories, magnitudes, scenarios,
  team affects, project metadata. No grammar change is required; the spec
  fixes the LLM-side consumer of facts that already exist.
- **The pipeline DAG and cache machinery** (`libterrain/src/pipeline.js`,
  `libterrain/src/nodes.js`, `ProseCache`). Stage topology, cache layout, and
  the `inspect` / `check` / `build` verbs are unchanged.
- **Prose template wording beyond the driver-context exposure.** Tone,
  length, phrasing, and template-versus-template differences are the
  template-author's call. The spec requires only that the driver context the
  schema names is reachable from the template, not how the template phrases
  it.
- **Online evaluation of generated output.** Verification of the LLM's actual
  prose output is by manual inspection of generated data after the change
  ships; the spec's success criteria are unit-test-verifiable properties of
  the contract and the prose-context entries, not properties of LLM output.
- **`outpost_markdown`, `guide_html`, project READMEs, org READMEs,**
  and other non-activity prose surfaces. They live in `prose-keys.js` today
  but are not "activity types" and are not the source of the asymmetry the
  spec targets.
- **New activity types.** The spec demonstrates that the contract supports
  extension; it does not add a sixth type.

## Notes

### Evidence pointers (for design)

- Asymmetry surface: `libraries/libsyntheticgen/src/engine/prose-keys.js`,
  `addSnapshotCommentKeys` (lines 120–140) vs `addWebhookProseKeys`
  (lines 207–238).
- Hand-wired composition: `libraries/libsyntheticgen/src/engine/activity.js`
  (sub-generator wiring around lines 73, 82, 98, 102),
  `libraries/libsyntheticgen/src/engine/prose-keys.js:147-198`
  (`collectProseKeys` if-chain),
  `libraries/libsyntheticrender/src/render/raw.js:17-27`
  (`renderRawDocuments` per-type list).
- Prompt-side consumer:
  `libraries/libsyntheticprose/src/prompts/prose-user.prompt.md`,
  `driverContext` block (currently fed by webhook keys only).
- DSL facts already in scope:
  `libraries/libsyntheticgen/src/dsl/parser-blocks.js` — `parseDxDrivers`,
  `parseScenario` (drivers with trajectory and magnitude per team affect).

### Handoff context for design

The design owner picks: (a) where the shared activity-type contract lives —
its own library, an existing one, or as a published interface; (b) whether
"activity type" stays the right unit of registration or whether the unit
should be finer (per stage) or coarser (per category); (c) how prompt
templates select per-type guidance once the prose-context schema is uniform;
(d) how the prose-context schema is named and validated. The spec fixes the
properties (one contract, one schema, uniform driver-context exposure,
single-registration extension) without prescribing the mechanism.
