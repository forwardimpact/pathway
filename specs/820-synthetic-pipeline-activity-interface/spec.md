# Spec 820 — Synthetic pipeline uniform activity interface

## Problem

The synthetic data pipeline that backs `fit-terrain` (one of the library
CLIs catalogued under **Gear**, the umbrella product Platform Builders hire
to evaluate and improve agents —
[JTBD § Evaluate and Improve Agents](../../JTBD.md#platform-builders-evaluate-and-improve-agents))
binds activity-data outputs to the pipeline output-by-output rather than
through a shared contract. The asymmetry is observable: the same DSL fact
that crosses into the LLM for one activity-data output is dropped on the
floor for another.

**Concrete evidence — comment prose drops compound driver impact.**
The DSL models `dx_drivers` per team per scenario with `trajectory` and
`magnitude`. Webhook prose (PR bodies and review bodies) passes the full
driver array into the LLM context; snapshot-comment prose passes only a
single driver scalar.

| Prose-bearing output  | File                                                           | Driver context shape passed to LLM                                                |
| --------------------- | -------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| PR body (webhook)     | `libraries/libsyntheticgen/src/engine/prose-keys.js:209-223`   | `drivers: wk.drivers` (full array of `{driver_id, trajectory, magnitude}`)        |
| Review body (webhook) | `libraries/libsyntheticgen/src/engine/prose-keys.js:225-238`   | `drivers: wk.drivers` (full array)                                                |
| Snapshot comment      | `libraries/libsyntheticgen/src/engine/prose-keys.js:120-140`   | `driver`, `direction`, `magnitude` (one driver only — no array, no other drivers) |

The asymmetry has two upstream sources, both of which the contract must
fix:

1. `libraries/libsyntheticgen/src/engine/activity-comments.js`,
   `collectAffectCandidates` (lines 39–60), keeps only the top driver per
   team affect (`drivers[0]`) and discards the rest before the comment-key
   ever reaches the prose-context builder.
2. `libraries/libsyntheticgen/src/engine/prose-keys.js`,
   `addSnapshotCommentKeys` (lines 120–140), constructs the LLM context
   with scalar `driver`/`direction`/`magnitude` fields where webhook key
   construction passes a `drivers` array. The prompt template at
   `libraries/libsyntheticprose/src/prompts/prose-user.prompt.md` has a
   conditional `driverContext` block that is populated for webhook keys and
   left empty for comment keys.

Result: when a comment is generated by a person on a team that the active
scenario marks as declining on `clear_direction`, `deep_work`, **and**
`code_review`, the LLM is told only about the single primary driver. Other
co-declining drivers — and the magnitudes that rank them — never reach the
prompt.

**Concrete evidence — no shared contract across activity-data outputs.**
The pipeline binds activity-data outputs (snapshot comments, the GitHub
webhook stream that fans out to PR-body and review-body prose plus push
events, GetDX scores, evidence, initiatives, scorecards, roster snapshots,
project teams) to four stages — generation, prose-context, rendering,
validation — by hand at three distinct call sites:

- **Activity composition** —
  `libraries/libsyntheticgen/src/engine/activity.js`,
  `generateActivity` (lines 57–106): each sub-generator is invoked
  individually and each output is named individually in the returned
  aggregate. Adding a new output requires editing both the call list and
  the return object.
- **Prose-context collection** —
  `libraries/libsyntheticgen/src/engine/prose-keys.js`,
  `collectProseKeys` (lines 147–198): a sequence of conditional blocks
  where the activity-prose branches (`commentKeys`, `webhookKeys` at lines
  184–195) sit alongside non-activity branches (`org_readme`, projects,
  `guide_html`, `outpost_markdown`).
- **Output rendering** —
  `libraries/libsyntheticrender/src/render/raw.js`,
  `renderRawDocuments` (lines 17–30): a list of per-output render
  functions, mixing activity-data outputs with non-activity outputs.

Validation in `libraries/libsyntheticrender/src/validate-activity.js` is
organized by *concern* (payload schemas, sender usernames, snapshot refs,
delivery-id uniqueness, …), not by activity-data output, so it is not a
fourth uniform call site today. The contract decides whether validation
becomes a per-output stage or stays cross-cutting alongside the registry.

The DSL is rich (drivers with trajectory and magnitude, scenarios,
narratives, phases, project topics, team membership, person level and
discipline) but each prose-bearing output chooses its own subset of that
information arbitrarily, and the choice is undocumented. The
comment-vs-webhook driver-context asymmetry above is one instance.

## Goal

Replace the ad-hoc, output-by-output binding of activity-data outputs to
the synthetic pipeline with a single uniform contract that every
activity-data output binds to. The contract names the pipeline stages an
output may participate in (deterministic generation; optionally
prose-context construction; optionally rendering; optionally validation)
and a single named shape for the DSL-derived domain context that crosses
into the LLM. As a consequence of the contract being applied uniformly,
snapshot-comment prose receives the same full driver context that webhook
prose already receives, and the asymmetry above disappears. Adding the
next activity-data output becomes a single registration of one
implementation, not a coordinated edit across multiple call sites.

## Scope (in)

- **The activity-data outputs the pipeline binds today.** Every output
  generated by `generateActivity` in
  `libraries/libsyntheticgen/src/engine/activity.js` (snapshot comments;
  the GitHub webhook stream for PR-body, review-body, and push prose;
  GetDX scores, evidence, initiatives, scorecards; roster snapshots;
  project teams). Each must be expressible through the same shared
  contract, participating in whichever pipeline stages apply to that
  output. Not every output bears prose; not every output has a per-output
  validator today.
- **The pipeline stages the contract names.** Generation; prose-context
  construction; output rendering; validation. The contract decides the
  exact relationship between validation and per-output registration (see
  Problem § validation note); generation, prose-context, and rendering
  are uniform across outputs that participate in them.
- **The prose-context schema.** A single named shape for the LLM-bound
  domain context, with explicitly enumerated fields covering at minimum
  the DSL facts available to any prose-bearing output: scenario, team
  identity, author role, the full team-affect driver array
  (`{driver_id, trajectory, magnitude}` per driver), and project
  topic/tone where the output relates to a project. Snapshot-comment
  prose-context entries must populate the full driver array, not a
  single-driver scalar.
- **The asymmetry's upstream source.**
  `activity-comments.js` `collectAffectCandidates` must carry the full
  team-affect driver array forward, not only the top driver, so that
  downstream prose-context construction has the array to populate.
- **The prompt-side consumer of that schema.** The user prompt at
  `libraries/libsyntheticprose/src/prompts/prose-user.prompt.md` (and any
  templates that share its surface) must render the schema's driver
  context for every prose-bearing output that supplies it, including
  snapshot comments.
- **Unit tests for the logic.** Per-output unit tests covering
  whichever stages of the contract the output binds to, including a
  snapshot-comment test where the author belongs to a team marked
  declining on multiple drivers.

## Scope (out)

- **The four-library boundary** (`libterrain`, `libsyntheticgen`,
  `libsyntheticprose`, `libsyntheticrender`). Whether the contract
  preserves, redraws, or eliminates the boundary is a design question.
- **The DSL grammar and parser.** The terrain DSL today already names
  every domain fact this spec needs.
- **The pipeline DAG and cache machinery** (`libterrain/src/pipeline.js`,
  `libterrain/src/nodes.js`, `ProseCache`). Stage topology, cache layout,
  and the `inspect` / `check` / `build` verbs are unchanged.
- **Prose template wording beyond driver-context exposure.** Tone, length,
  phrasing, and template-versus-template differences are the
  template-author's call. The hardcoded driver-name list inside the
  template's `{{#driverContext}}` block stays untouched unless the schema
  change makes it incorrect.
- **Online evaluation of LLM output.** Verification of the LLM's actual
  prose output is by post-merge manual inspection, not by an automated
  test. The spec's success criteria are testable properties of the
  contract and the prose-context entries, not of LLM output.
- **Non-activity prose surfaces.** `org_readme`, project READMEs,
  `guide_html`, `outpost_markdown` are not activity-data outputs and stay
  on their existing path. The contract does not consume them and the
  call-site shims continue to dispatch them alongside the registry.
- **New activity-data outputs.** The spec demonstrates the contract
  supports extension; it does not add a new output.

## Success criteria

| #   | Claim                                                                                                                                                                                                       | Verification                                                                                                                                                                                                                            |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | A single named contract names every pipeline stage an activity-data output may participate in, and every in-scope output binds to it.                                                                       | Static inspection: one named module/file enumerates the in-scope activity-data outputs and exposes the same named members per output (one member per stage that output participates in).                                                |
| 2   | The activity composition, prose-context collection, and output rendering call sites no longer name individual activity-data outputs in their own bodies.                                                    | Static inspection: no in-scope activity-data output identifier appears as a literal at the three call sites named in Problem (`generateActivity`, `collectProseKeys` activity branches, `renderRawDocuments` activity branches).        |
| 3   | A snapshot-comment prose-context entry, for an author on a team that the scenario marks as declining on at least two drivers, exposes the same multi-driver context that webhook prose-context entries do. | Unit test on the prose-context construction: given a fixture entity graph with a multi-driver-declining team, the resulting prose-context entry for that comment carries an array with one entry per declining driver from the affect. |
| 4   | The prompt rendered for a prose-bearing output is a function of its prose-context entry alone, with no per-output branching at the prompt construction site.                                                | Unit test on prompt construction: given equivalent prose-context entries (same drivers array, scenario, role) for a webhook key and a comment key, the rendered prompts' driver-context blocks are equal.                               |
| 5   | Each in-scope activity-data output has unit-test coverage for whichever pipeline stages it binds to under the contract; at least one prose-bearing output's tests run against multi-driver-declining input. | Test inventory: per output, named tests cover each stage the output participates in; the snapshot-comment tests include a multi-driver-declining fixture.                                                                               |
| 6   | The post-refactor pipeline produces the same set of generated output file paths as the pre-refactor pipeline for an unchanged DSL input.                                                                    | Replay `fit-terrain build` on a representative DSL fixture pre- and post-refactor; the set of file paths produced is identical. (Comment file *contents* will change because the prose-context entries change; this is expected.)       |

### Findings panel disposition

This spec was reviewed by a panel of 3 product reviewers and 3 technical
reviewers; consensus blocker/high/medium findings have been addressed in
this revision. Notable changes from the prior draft: dropped the
"five activity types" framing as inconsistent with the code (PRs/reviews
are webhook event types, not peers; "GetDX webhooks" do not exist;
commits do not have prose); restated success criteria #1, #2, #6 as
observable properties without prescribing registry/iteration mechanisms;
fixed criterion #7 to drop the manual half; explicitly scoped in
`collectAffectCandidates` (the upstream source of the comment asymmetry);
acknowledged that validation is concern-keyed today and the contract
decides how it participates.
