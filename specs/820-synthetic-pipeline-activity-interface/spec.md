# Spec 820 — Synthetic pipeline uniform activity interface

## Problem

The synthetic data pipeline that backs `fit-terrain` (one of the library
CLIs catalogued under **Gear**, the umbrella product Platform Builders
hire to evaluate and improve agents —
[JTBD § Evaluate and Improve Agents](../../JTBD.md#platform-builders-evaluate-and-improve-agents))
binds prose-bearing activity outputs to the pipeline output-by-output
rather than through a shared contract. The asymmetry is observable: the
same DSL fact that crosses into the LLM for one prose-bearing output is
dropped on the floor for another.

### Comment prose drops compound driver impact

The DSL models `dx_drivers` per team per scenario with `trajectory` and
`magnitude`. Webhook prose (PR bodies and review bodies) passes the full
driver array into the LLM context; snapshot-comment prose passes only a
single driver scalar.

| Prose-bearing output  | File                                                           | Driver context shape passed to LLM                                                |
| --------------------- | -------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| PR body (webhook)     | `libraries/libsyntheticgen/src/engine/prose-keys.js:209-223`   | `drivers: wk.drivers` (full array of `{driver_id, trajectory, magnitude}`)        |
| Review body (webhook) | `libraries/libsyntheticgen/src/engine/prose-keys.js:225-238`   | `drivers: wk.drivers` (full array)                                                |
| Snapshot comment      | `libraries/libsyntheticgen/src/engine/prose-keys.js:120-140`   | `driver`, `direction`, `magnitude` (one driver only — no array, no other drivers) |

The asymmetry has two upstream sources:

1. `libraries/libsyntheticgen/src/engine/activity-comments.js`,
   `collectAffectCandidates` (lines 39–60), keeps only the top driver per
   team affect (`drivers[0]`) and discards the rest before the
   comment-key reaches the prose-context builder.
2. `libraries/libsyntheticgen/src/engine/prose-keys.js`,
   `addSnapshotCommentKeys` (lines 120–140), constructs the LLM context
   with scalar `driver`/`direction`/`magnitude` fields where webhook key
   construction passes a `drivers` array. The prompt template at
   `libraries/libsyntheticprose/src/prompts/prose-user.prompt.md` has a
   conditional `driverContext` block that is populated for webhook keys
   and left empty for comment keys.

When a comment is generated by a person on a team that the active
scenario marks as declining on multiple drivers, the LLM is told only
about the single primary driver; the magnitudes that rank the others
never reach the prompt.

### No shared contract across prose-bearing outputs

The pipeline binds prose-bearing outputs (snapshot comments and the
GitHub webhook stream that fans out to PR-body and review-body prose) by
hand at three distinct call sites:

- **Activity composition** —
  `libraries/libsyntheticgen/src/engine/activity.js`,
  `generateActivity` (lines 57–106): each sub-generator is invoked
  individually and each output is named individually in the returned
  aggregate.
- **Prose-context collection** —
  `libraries/libsyntheticgen/src/engine/prose-keys.js`,
  `collectProseKeys` (lines 147–198): a sequence of conditional blocks
  where the activity-prose branches (`commentKeys`, `webhookKeys` at
  lines 184–195) sit alongside non-activity branches.
- **Output rendering** —
  `libraries/libsyntheticrender/src/render/raw.js`,
  `renderRawDocuments` (lines 17–30): a list of per-output render
  functions.

None of the three lists agrees on a name, a signature, or an order.
Adding a new prose-bearing output requires editing all three lists plus
the prompt-side consumer. The DSL is rich (drivers with trajectory and
magnitude, scenarios, narratives, phases, project topics, team
membership, person level and discipline) but each prose-bearing output
chooses its own subset of that information arbitrarily, and the choice
is undocumented. The comment-vs-webhook driver-context asymmetry above
is one instance.

## Goal

Replace the ad-hoc, output-by-output binding of prose-bearing activity
outputs to the synthetic pipeline with a uniform contract that every
prose-bearing output binds to. The contract names the pipeline stages a
prose-bearing output participates in (deterministic generation,
prose-context construction, output rendering) and a single named shape
for the DSL-derived domain context that crosses into the LLM. As a
consequence of the contract being applied uniformly, snapshot-comment
prose receives the same full driver context that webhook prose already
receives, and the asymmetry above disappears. Adding the next
prose-bearing output becomes a localized addition, not a coordinated
edit across multiple call sites.

## Scope (in)

- **Prose-bearing activity outputs.** Snapshot comments and the GitHub
  webhook stream (which produces PR-body prose, review-body prose, and
  push events alongside). These are the outputs that exhibit the
  driver-context asymmetry and that motivate the contract.
- **The three pipeline call sites named in Problem § "No shared
  contract".** Activity composition, prose-context collection, and
  output rendering must dispatch the in-scope prose-bearing outputs
  through one shared mechanism rather than naming each output
  individually.
- **The prose-context schema.** A single named shape for the LLM-bound
  domain context, with explicitly enumerated fields covering the DSL
  facts available to any prose-bearing output: scenario, team identity,
  author role, the full team-affect driver array
  (`{driver_id, trajectory, magnitude}` per driver). Snapshot-comment
  prose-context entries must populate the full driver array, not a
  single-driver scalar.
- **The asymmetry's upstream source.**
  `activity-comments.js` `collectAffectCandidates` carries only the top
  driver forward; the contract's data flow must carry the full
  team-affect driver array end-to-end so the snapshot-comment
  prose-context entry can populate it.
- **The prompt-side consumer.** `prose-user.prompt.md` (the only
  template that consumes activity prose-context entries today) must
  render the schema's driver context for every in-scope prose-bearing
  output that supplies it, including snapshot comments. The prompt
  builder at `generator.js` `#buildPrompt` (lines 169–190) is in scope
  for whatever change makes the rendered prompt a function of the
  prose-context entry alone.
- **Per-output unit tests** for the contract-bound logic, including a
  snapshot-comment test where the author belongs to a team marked
  declining on multiple drivers.

## Scope (out)

- **Non-prose-bearing activity outputs** (`scores`, `evidence`,
  `initiatives`, `scorecards`, `rosterSnapshots`, `projectTeams`,
  `roster`, `activityTeams`, `snapshots`, the `webhooks` event stream
  itself when consumed for rendering rather than prose). They are
  generated and rendered today, but they do not exhibit the asymmetry,
  do not consume the prose-context schema, and are not the source of
  the call-site duplication this spec targets. They stay where they
  are; the contract does not bind them.
- **Validation.** `libraries/libsyntheticrender/src/validate-activity.js`
  is concern-keyed today (payload schemas, sender usernames, snapshot
  refs, delivery-id uniqueness, …), not per-output. The contract does
  not subsume validation; concern-keyed validators stay as they are,
  with whatever minimal updates are needed to read post-refactor field
  names.
- **The four-library boundary** (`libterrain`, `libsyntheticgen`,
  `libsyntheticprose`, `libsyntheticrender`). Whether the contract
  preserves, redraws, or eliminates the boundary is a design question.
- **Non-activity prose surfaces.** `org_readme`, project READMEs,
  `guide_html`, `outpost_markdown`. The call-site shims continue to
  dispatch them inline alongside the contract-driven dispatch.
- **The DSL grammar and parser.** No grammar change is required.
- **The pipeline DAG and cache machinery** (`libterrain/src/pipeline.js`,
  `libterrain/src/nodes.js`, `libsyntheticprose/.../ProseCache`).
- **Prose template wording beyond driver-context exposure.** Tone,
  length, phrasing, and the hardcoded driver-name list inside the
  template's `{{#driverContext}}` block stay untouched unless the
  schema change makes them incorrect.
- **Online evaluation of LLM output.** Verification of the LLM's actual
  prose output is by post-merge manual inspection, not by an automated
  test.
- **New prose-bearing outputs.** The spec demonstrates the contract
  supports extension; it does not add a new output.

## Success criteria

## Notes — evidence pointers (for design)

- Asymmetry surface (both ends):
  `activity-comments.js` `collectAffectCandidates` (39–60, top-driver
  collection) and `prose-keys.js` `addSnapshotCommentKeys` (120–140) vs
  `addWebhookProseKeys` (207–238).
- Hand-wired call sites: `activity.js` `generateActivity` (57–106),
  `prose-keys.js` `collectProseKeys` (147–198),
  `raw.js` `renderRawDocuments` (17–30).
- Validation surface (concern-keyed, out of scope for the contract):
  `libsyntheticrender/src/validate-activity.js`.
- Prompt-side consumer:
  `libsyntheticprose/src/prompts/prose-user.prompt.md` `driverContext`
  block, built by `libsyntheticprose/src/engine/generator.js`
  `#buildPrompt` (169–190).
- DSL facts already in scope:
  `libsyntheticgen/src/dsl/parser-blocks.js` `parseDxDrivers`,
  `parseScenario`.
