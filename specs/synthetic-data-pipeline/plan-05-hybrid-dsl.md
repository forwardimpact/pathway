# Plan 05 — Hybrid DSL with Tiered Generation

> A custom domain-specific language (DSL) defines the data universe
> declaratively; a tiered execution engine uses deterministic generators for
> structure and LLM calls only where natural language is required, with the
> option to run fully offline using cached LLM output.

## Approach

Define a DSL that captures the full data universe — entities, relationships,
scenarios, signal parameters — in a single composable specification. A tiered
execution engine processes this specification:

- **Tier 0 (deterministic):** Entity graphs, relationships, activity data,
  signal curves — pure functions, seeded PRNG, no LLM
- **Tier 1 (LLM-assisted):** Prose fields (descriptions, evidence, articles)
  — generated by any LLM backend (Claude API, local models, or cached)
- **Tier 2 (cached):** Previously generated Tier 1 output stored in a cache
  file, enabling fully offline/deterministic runs after initial generation

The key insight: separate what *must* be natural language from what can be
computed. Only ~15% of output tokens require LLM generation; the rest is
structural.

## Architecture

```
universe.dsl ──► DSL Parser
                    │
                    ├── Entity Graph Builder (Tier 0)
                    │    └── Deterministic: people, orgs, projects, relations
                    │
                    ├── Signal Engine (Tier 0)
                    │    └── Deterministic: curves, events, scores
                    │
                    ├── Prose Generator (Tier 1 or Tier 2)
                    │    ├── Claude API / Ollama / llama.cpp
                    │    └── Cache file (.prose-cache.json)
                    │
                    └── Renderers
                         ├── HTML Renderer (microdata)
                         ├── YAML Renderer (framework)
                         ├── Table Renderer (JSON/CSV)
                         └── Markdown Renderer (personal)
                    │
                    ▼
               Validator Suite
                    │
               Monorepo Export
```

### DSL Parser

The DSL parser is a simple recursive-descent parser that produces an AST:

```javascript
// scripts/generate/dsl/parser.js

/**
 * Parse universe DSL into an AST.
 * @param {string} source - DSL source text
 * @returns {UniverseAST}
 */
export function parseUniverse(source) {
  const tokens = tokenize(source)
  const parser = new Parser(tokens)
  return parser.universe()
}

// AST node types
// UniverseNode { name, orgs, departments, teams, people, projects, scenarios, framework, content }
// DepartmentNode { id, name, parent, headcount, teams }
// TeamNode { id, name, size, manager, repos }
// ScenarioNode { id, name, timerange, affects, prose }
// ProseNode { topic, tone, length, vocabulary_standard }
// ... etc
```

### Tier 1 — LLM-Assisted Prose

Only invoked for AST nodes tagged with `prose`:

```javascript
// scripts/generate/engine/tier1.js

export class ProseEngine {
  constructor(llmClient, cache) {
    this.llm = llmClient
    this.cache = cache
  }

  /**
   * Generate prose for a node, checking cache first.
   * @param {ProseNode} node - AST prose node
   * @param {object} context - Entity context for the prose
   * @returns {string} Generated prose
   */
  async generateProse(node, context) {
    const cacheKey = this.computeCacheKey(node, context)

    // Tier 2: return cached if available
    if (this.cache.has(cacheKey)) {
      return this.cache.get(cacheKey)
    }

    // Tier 1: generate via LLM
    const prompt = this.buildPrompt(node, context)
    const result = await this.llm.complete(prompt)

    // Cache for future deterministic runs
    this.cache.set(cacheKey, result)
    return result
  }

  buildPrompt(node, context) {
    const vocabularyContext = node.vocabulary_standard === 'agents_md'
      ? VOCABULARY_STANDARDS
      : ''

    return `Generate ${node.length} of ${node.tone || 'technical'} prose about:
Topic: ${node.topic}
Context: ${JSON.stringify(context)}
${vocabularyContext}
Output the text only, no explanations.`
  }

  computeCacheKey(node, context) {
    return hashValues(node.topic, node.tone, node.length, JSON.stringify(context))
  }
}
```

When running with `--offline` or `--cached`, the pipeline uses only cached
prose and fails fast if any cache miss occurs:

```sh

# Subsequent runs: fully deterministic from cache

# CI runs: fail if cache is stale
npx fit-map generate --cached --strict
```

```javascript
export class HTMLRenderer {
  /**
   * Render entities to HTML microdata files.

      this.renderRolesHTML(entities.people))

    for (const articleTopic of entities.content.articles) {
      files.set(`articles-${articleTopic.slug}.html`,
        this.renderArticlesHTML(articleTopic, entities, prose))

    // ... all 22 HTML files
```

### Markdown Renderer

```javascript
// scripts/generate/renderers/markdown.js

export class MarkdownRenderer {
  render(personas, activity, prose) {
    const files = new Map()

    for (const persona of personas) {
      const dir = `persona-${persona.name.toLowerCase()}`

      // Weekly briefings
      for (const briefing of persona.briefings) {
        files.set(
          `${dir}/Briefings/${briefing.date}.md`,
          this.renderBriefing(briefing, persona, prose)
        )
      }

      // Knowledge notes
      for (const note of persona.notes) {
        files.set(
          `${dir}/${note.category}/${note.title}.md`,
          this.renderNote(note, persona, prose)
        )
      }
    }

    return files
  }
}
```

```javascript
export class UniverseValidator {
  constructor(entities, framework, activity, html, personal) {
    this.entities = entities
    this.framework = framework
    this.activity = activity
    this.html = html
    this.personal = personal
  }

  validate() {
    const checks = [
      this.checkPeopleCoverage(),
      this.checkFrameworkValidity(),
      this.checkGitHubCorrelation(),
      this.checkSurveyCorrelation(),
      this.checkEvidenceProficiency(),
      this.checkHTMLIRIs(),
      this.checkSelfAssessments(),
      this.checkBasecampReferences(),
      this.checkFitMapValidate(),
    ]

    const failures = checks.filter(c => !c.passed)
    return {
      passed: failures.length === 0,
      total: checks.length,
      failures,
    }
  }

  async checkFitMapValidate() {
    const result = await exec('npx fit-map validate')
    return { name: 'fit-map validate', passed: result.exitCode === 0 }
  }
}
```

- Documentation

## Strengths

- **Best of both worlds**: Deterministic structure + LLM prose quality
- **Cache makes it deterministic**: After first LLM run, all subsequent runs
  are fully reproducible from cache
- **CI-friendly**: Cached mode runs in seconds, fails fast on stale cache
- **DSL is self-documenting**: The universe file is a readable specification
  of the entire synthetic data universe
- **LLM-agnostic**: Tier 1 works with any LLM backend
- **Minimal LLM usage**: Only ~15% of output requires LLM generation,
  reducing cost and time
- **Pure Node.js**: No Python dependency (unlike Plans 02, 03)
- **Composable**: New entities/scenarios are added by extending the DSL file,
  not by writing new templates or code

## Weaknesses

- **Custom DSL maintenance**: The DSL parser is bespoke code that must be
  maintained; changes to the data model require DSL grammar updates
- **Two-run workflow**: First run requires LLM access; only subsequent runs
  are fully offline — new content additions always need an LLM run
- **Cache staleness**: If the DSL changes, cache keys change, requiring
  regeneration of affected prose
- **DSL learning curve**: Contributors must learn the DSL syntax to modify
  the universe definition
- **Parser complexity**: A recursive-descent parser, while simple, is still
  ~500 lines of code to maintain
