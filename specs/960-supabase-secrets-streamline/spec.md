# Spec 960 â€” Streamline Supabase Secrets and JWT Authentication

## Problem

A single Supabase instance backs every product in the monorepo (Map writes
ingestion data; Landmark reads via per-caller JWT; Summit reads via service
role; libstorage talks to Supabase Storage; Map edge functions consume the
same instance from inside Deno). Despite this, the codebase treats Supabase
credentials as if seven independent variables, generated by two different
scripts, were the right shape:

```
JWT_SECRET                       â€” generic HS256 secret (used by storage-api)
SUPABASE_SERVICE_ROLE_KEY        â€” generated by env-storage.js
MAP_SUPABASE_URL                 â€” read by 10+ call sites
MAP_SUPABASE_SERVICE_ROLE_KEY    â€” generated by env-secrets.js AND env-storage.js
MAP_SUPABASE_ANON_KEY            â€” generated by env-storage.js
MAP_SUPABASE_JWT_SECRET          â€” generated by nothing automated
MAP_SUPABASE_DB_PORT             â€” set in three .env examples; no consumer in source
```

The `MAP_` prefix presupposes per-product Supabase projects, but Landmark and
Summit reach into `MAP_SUPABASE_*` directly â€” there is no `LANDMARK_SUPABASE_*`
or `SUMMIT_SUPABASE_*`. The presupposition is false on the ground.

Three concrete defects fall out of this layout:

1. **Two scripts write `MAP_SUPABASE_SERVICE_ROLE_KEY` to two different files
   with two different signing inputs.**
   `scripts/env-secrets.js:76` writes the variable to `.env`, signed against
   a hardcoded demo string (`"super-secret-jwt-token-with-at-least-32-characters-long"`)
   because the local Supabase CLI started by `fit-map activity start` rejects
   anything else. `scripts/env-storage.js:81-85` writes the same variable name
   to `.env.storage.supabase`, signed against the freshly-generated `JWT_SECRET`.
   `docker-compose.yml` lines 175, 213, 236, and 261 pass both files via
   `env_file: [.env, .env.storage.supabase]`, so the `.env.storage.supabase`
   value wins inside containers; outside docker, the `.env` value wins. The
   operator then pastes a *third* value (from `fit-map activity start` output)
   into the shell to make the local CLI flow work, overriding whichever file
   value was loaded. The three values never agree, and which one is active
   depends on whether the process runs inside docker, outside docker via
   `bun`, or via a shell that pasted the activity-start export. See
   `scripts/env-secrets.js:32-83`, `scripts/env-storage.js:13-99`, and
   `products/map/src/commands/activity.js:23-39`.

2. **`MAP_SUPABASE_JWT_SECRET` has no automated path at all.** Three code
   paths demand it â€” `fit-map auth issue`
   (`products/map/src/commands/auth-issue.js:53-60`), Landmark's identity HMAC
   verify (`products/landmark/src/lib/identity.js:71-84`), and the
   `signTestToken` test helper
   (`products/landmark/test/lib/sign-test-token.js:14-18`) â€” but neither
   `env-secrets.js` nor `env-storage.js` writes it. The only source is the
   operator copy/pasting from `fit-map activity start` output. If the operator
   forgets, `auth issue` errors and Landmark's identity check silently
   degrades from HMAC-verify to shape-only verify
   (`identity.js:71` is gated on `if (env.MAP_SUPABASE_JWT_SECRET)`).

3. **`SUPABASE_SERVICE_ROLE_KEY` and `MAP_SUPABASE_SERVICE_ROLE_KEY` are
   bit-identical duplicates.** `scripts/env-storage.js:37-86` generates both
   from the same `JWT_SECRET` with the same payload (`role: service_role`,
   `iss: supabase`, 10-year `exp`). Both reach the same Postgres via
   different `docker-compose.yml` env-passthroughs (lines 177 vs 241). Two
   names, one key.

Separately, **no central facility exists in `libconfig` for Supabase
variables.** `Config.#CREDENTIAL_KEYS`
(`libraries/libconfig/src/config.js:48-53`) registers
`{ANTHROPIC_API_KEY, GH_TOKEN, GITHUB_TOKEN, MCP_TOKEN}` and provides typed
accessors (`anthropicToken()`, `ghToken()`, `mcpToken()`). Supabase
credentials are not registered â€” they leak onto `process.env` and into child
process inheritance â€” and every consumer reads `process.env.MAP_SUPABASE_*`
directly. Each call site re-implements the same "is it set? throw a custom
error pointing the operator at `fit-map activity start`" pattern. The
duplication lives across these files, with line numbers as of `main`:

|---|---|
| `services/map/server.js:13-20` | `MAP_SUPABASE_URL`, `MAP_SUPABASE_SERVICE_ROLE_KEY`, plus dead `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` fallback |
| `libraries/libterrain/src/cli-helpers.js:59-71` | `MAP_SUPABASE_URL`, `MAP_SUPABASE_SERVICE_ROLE_KEY` |
| `libraries/libstorage/src/index.js:199-207` | `SUPABASE_SERVICE_ROLE_KEY`, `AWS_ENDPOINT_URL` |
| `products/map/src/lib/client.js:13-15` | `MAP_SUPABASE_URL`, `MAP_SUPABASE_SERVICE_ROLE_KEY` |
| `products/map/src/commands/auth-issue.js:53` | `MAP_SUPABASE_JWT_SECRET` |
| `products/landmark/src/lib/identity.js:71-99` | `MAP_SUPABASE_JWT_SECRET`, `MAP_SUPABASE_URL`, `MAP_SUPABASE_ANON_KEY` |
| `products/landmark/src/lib/supabase.js:32-33` | `MAP_SUPABASE_URL`, `MAP_SUPABASE_ANON_KEY` |

The blast radius is every internal contributor who runs `just env-setup`
followed by `fit-map activity start`: their `.env` does not match their
shell, their shell does not match their Supabase instance, and the
inconsistency surfaces as confusing auth failures at the Landmark or Map
boundary. Net cost is hours of debugging per onboarding contributor and a
recurring footgun for everyone who re-runs `env-setup` after a database
reset.

## Scope

This is a **clean break**. No backward-compatibility shims, no deprecated
aliases, no `MAP_SUPABASE_*` fallback chain, no transitional period. The new
names land and the old names are deleted in the same change.

### In scope

| Component | What changes |
|---|---|
| Environment-variable surface | The seven existing variables collapse to four canonical, unprefixed names: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_JWT_SECRET`. `JWT_SECRET`, `MAP_SUPABASE_URL`, `MAP_SUPABASE_ANON_KEY`, `MAP_SUPABASE_SERVICE_ROLE_KEY`, `MAP_SUPABASE_JWT_SECRET`, `MAP_SUPABASE_DB_PORT`, and the `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` fallback branches are removed from every example file, every script, every product, every service, every library, every test, every docker-compose service, and every documentation page. | Single signing secret | `SUPABASE_JWT_SECRET` is the sole HS256 signing secret for the Supabase instance. ### Out of scope, deferred

- **Multi-instance Supabase support.** The premise of this spec is one
  Supabase instance per monorepo. Re-introducing a prefix to disambiguate
  between two instances is out of scope. If a future change ever requires
  multiple instances, that future spec re-introduces a prefix and pays the
  migration cost.
- **Replacing or simplifying Supabase Auth itself.** The HS256/JWT contract
  with Supabase Auth, the magic-link login flow, the
  `LANDMARK_AUTH_TOKEN` engineer-side token, and the service-role write path
  for Map ingestion are unchanged. v1 renames the env vars and centralizes
  reads; the underlying auth model is unchanged.
- **Moving the local Supabase stack off the `supabase` CLI.** If the design
  chooses `supabase start --jwt-secret <ours>` rather than the docker-compose
  `map-supabase` profile, the existing CLI-based local workflow remains;
  switching everyone to docker-compose is out of scope.
- **Adding Supabase secrets to `gitleaks` or secret-scan rules.** The
  `.gitleaks.toml` rules are unchanged by this spec.
- **Consolidating storage credentials** (`AWS_ACCESS_KEY_ID`,
  `AWS_SECRET_ACCESS_KEY`, `S3_BUCKET_NAME`, `AWS_ENDPOINT_URL`) into the
  same `libconfig` accessor pattern. The storage variables are already
  scoped sensibly and have their own consumer at `libstorage`; their cleanup
  is a separate spec if anyone wants it. Only the `SUPABASE_SERVICE_ROLE_KEY`
  read in `libstorage/src/index.js:201` is migrated here, because it shares
  the Supabase JWT-secret signing chain.
- **`config/config.json` defaults for Supabase.** The accessors are env-only
  in v1, matching `mcpToken()` and `anthropicToken()`. Adding a config.json
  override path is a separate spec.
